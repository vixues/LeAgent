# 一、Agent 基础概念

## 1.1 什么是 AI Agent？

AI Agent 是能够围绕目标感知上下文、作出决策、执行动作并依据结果继续调整的自治或半自治系统。它通常以 LLM 作为推理与决策核心，以工具连接外部世界，并通过循环把一次文本生成扩展为多步任务执行。

在 LeAgent 中，所有 Agent 轮次统一进入 `leagent.sdk.kernel.loop.run_loop`，再由 `QueryEngine` 与 `ToolExecutor` 完成 think-act 循环；会话、检查点和工作流状态由不同的持久化组件分别负责。

## 1.2 Agent 和 Chatbot 的区别？

Chatbot 的主要产物是回复，典型路径是“输入—生成文本”；Agent 的主要产物是目标状态变化，路径是“理解目标—规划—调用工具—观察结果—继续决策”。Chatbot 可以有记忆，但未必有行动能力；Agent 通常需要工具、权限、预算和停止条件。

LeAgent 的聊天入口虽然通过 SSE 展示结果，但内部仍进入统一 Agent 内核，因此既能输出文本，也能执行工具、暂停等待审批、恢复检查点或委派子 Agent。

## 1.3 Agent 和 Workflow 的区别？

Agent 由模型在运行时决定下一步，适合开放目标和不确定环境；Workflow 预先定义节点、依赖和控制流，适合稳定、可审计、可重复的过程。前者灵活但方差较大，后者可预测但难覆盖未知分支。工程上常把 Agent 放进 Workflow 节点，或让 Agent 生成、选择和运行 Workflow。

LeAgent 同时提供 Agent think-act 内核与 `WorkflowExecutor` DAG 引擎。保存流程、定时任务和聊天步骤卡最终复用工作流执行器，而 `Agent.<name>` 又可作为一等工作流节点。

## 1.4 Agent 为什么比 Prompt 更强？

Prompt 只影响一次或有限轮次的生成行为；Agent 还引入状态、工具、反馈、预算、记忆和控制循环。它能把中间结果重新放入上下文，根据真实执行结果纠错，而不是仅凭首次生成猜测答案。

强并不等于始终更好：多轮调用也会增加延迟、费用和故障面。LeAgent 因此用 `AgentDefinition` 集中声明工具、模型、记忆和运行预算，而不是把全部能力寄托在一段提示词里。

## 1.5 Agent 的核心组成有哪些？

通常包括：目标与指令、LLM 或其他决策器、上下文与状态、规划机制、工具及执行器、记忆、反馈循环、安全权限、预算与停止条件、可观测性。生产系统还需要持久化、恢复、错误处理和人工审批。

LeAgent 将这些职责拆为 `AgentDefinition`、`RuntimeContext` 和 `AgentRuntime` 三类契约；运行时统一接入 LLM、`ContextManager`、`AgentMemory`、`ToolRegistry`、`ToolExecutor`、hooks 和 `CheckpointStore`。

## 1.6 Agent Loop 是什么？

Agent Loop 是反复执行“观察当前状态—推理下一步—采取动作—读取结果—更新状态”的控制循环。若模型直接给出最终答案、达到预算、被取消、等待用户输入或发生不可恢复错误，循环结束。

LeAgent 的 `query()` 每轮把模型输出中的 tool calls 交给执行器，将工具结果以 `tool` 消息追加到历史，再进入下一轮；`max_turns` 和 `max_tool_calls_per_turn` 分别限制总轮次与单轮工具数。

## 1.7 ReAct 是什么？

ReAct 即 Reasoning and Acting，把推理与行动交替组织：模型基于当前观察形成下一步判断，调用工具获得新观察，再继续推理。其关键不是必须暴露完整思维链，而是让决策与外部反馈形成闭环。

工程实现通常只保留可审计的动作、工具参数、结果和简短决策摘要，不应依赖输出私有思维过程。LeAgent 的流式事件和 transcript 记录工具调用及结果，核心循环负责交替推进。

## 1.8 ReAct 为什么有效？

它把大问题拆成可验证的小步骤，并用搜索、计算、数据库等真实结果修正模型先验，减少“一次生成到底”的累积猜测。每次行动也缩小了下一步决策空间。

但 ReAct 仍可能选错工具或被错误观察带偏，所以需要 schema 校验、权限、重试边界与轮次上限。LeAgent 在执行前校验参数和审批需求，在执行后把结构化成功或错误结果送回模型。

## 1.9 Tool Use 的本质是什么？

Tool Use 的本质是受约束的能力委托：模型生成“调用哪个能力及其结构化参数”，确定性代码负责校验、授权、执行和返回结果。模型并未直接访问文件、网络或数据库，真正副作用发生在受控执行层。

LeAgent 中工具通过 `BaseTool` 声明名称、描述和 JSON Schema，由 `ToolRegistry` 暴露给模型，由 `ToolExecutor` 统一处理查找、权限、超时、重试、沙箱和结果封装。

## 1.10 Agent 为什么需要 Planning？

Planning 用于显式建立子目标、依赖、资源与完成标准，避免模型只追逐眼前一步。它对长任务、不可逆操作、步骤依赖和高成本调用尤其重要，也便于用户提前审阅。

Planning 不应变成僵化的长清单；环境变化后应允许重规划。LeAgent 支持计划模式，并在执行器层阻止计划期间的副作用工具，只允许只读工具和少数计划控制工具，退出计划模式可要求用户确认。

## 1.11 Agent 的定义是什么？

工程上的 Agent 定义应是可复现的声明式契约，而不只是“一个 Prompt”。它至少应确定身份与指令、可用工具、模型策略、记忆策略、上下文组装方式、运行预算及可委派对象。

LeAgent 的 `AgentDefinition` 正是这一真源：包含 `prompt_variant`、`context_recipe`、`ToolPolicy`、`ModelPolicy`、`MemoryPolicy`、`runtime_profile`、`max_turns` 和子 Agent 等字段；`AgentRuntime` 再将其物化为执行配置。

## 1.12 Agent 是否一定需要 LLM？

不一定。只要系统能感知、决策、行动并根据反馈调整，就可以由规则、搜索、强化学习策略或传统规划器构成 Agent。LLM 的优势是处理自然语言和开放任务，但不是概念上的必要条件。

当前 LeAgent 的通用 think-act 路径以 LLM 作为主要决策器；对于控制流确定的任务，则更适合直接使用其 DAG Workflow，而不是人为增加 LLM 决策。

## 1.13 什么场景适合 Agent？

适合目标明确但路径不固定、需要跨工具收集信息、允许迭代纠错且错误可控的任务，例如研究汇总、代码辅助、文档处理和复杂办公协作。判断标准是运行时决策的收益能否覆盖额外成本与风险。

LeAgent 可通过领域 `AgentDefinition` 缩小工具和模型策略，也可将 Agent 嵌入工作流，只把真正不确定的节点交给模型。

## 1.14 什么场景不适合 Agent？

不适合规则稳定、输入输出严格、低延迟高吞吐、强合规或错误代价极高且无法人工复核的任务，例如简单字段转换、核心账务记账和硬实时控制。此时普通函数、规则引擎或确定性 Workflow 更可靠。

在 LeAgent 中，已知步骤应优先落到 `WorkflowDocument` 与 `WorkflowExecutor`；有副作用的 Agent 工具还应配合 deny/allow 策略、沙箱和审批。

## 1.15 Agent 为什么容易不稳定？

因为它把概率生成嵌入反馈循环：一次细小的工具选择或参数差异，会改变观察结果和后续上下文，最终形成完全不同的轨迹。模型版本、采样参数、上下文截断、外部服务和并发状态也会引入波动。

LeAgent 通过低温度策略、严格工具契约、统一上下文组装、检查点和运行预算降低方差，但无法把开放式 Agent 变成完全确定的程序。

## 1.16 Agent 的不可预测性来自哪里？

主要来自模型采样与近似推理、自然语言歧义、提示词和历史消息的上下文敏感性、工具返回的动态数据、并发与网络时序，以及多轮误差放大。即使 temperature 为 0，服务端实现和并列 token 也不保证跨版本绝对复现。

LeAgent 用 `run_id`、`parent_run_id`、事件流、工具调用记录和持久化 transcript 提供轨迹级可观测性，使问题可以按一次真实运行追溯，而不是只比较最终文本。

## 1.17 Agent 的主要失败模式有哪些？

常见失败包括：误解目标、错误规划、选错或漏用工具、参数不合法、幻觉工具结果、重复调用、过早停止、无限循环、上下文污染、权限越界、错误恢复失败和成本失控。多 Agent 系统还会出现委派错误与状态合并问题。

LeAgent 对未知参数、条件必填字段和 JSON Schema 做执行前校验；对工具设置超时与有限重试，对循环设置轮次及单轮调用上限，并对敏感副作用提供审批暂停。

## 1.18 为什么 Agent 比传统软件更难测试？

传统软件通常能对固定输入断言固定输出；Agent 的正确性更像轨迹质量：最终答案可能不同但都正确，或答案看似正确但中间执行了危险动作。测试还依赖模型、外部工具、时间和上下文状态。

因此需要分层测试：工具与 schema 做确定性单测，循环与事件做契约测试，完整任务做成功率、成本、延迟和安全评测，并保存失败轨迹回归。LeAgent 已对 runtime、kernel checkpoint、SSE 线缆契约和工作流 Agent 节点分别测试。

## 1.19 Agent 的上限取决于什么？

取决于基础模型的推理与指令遵循能力、工具的覆盖面和可靠性、上下文与记忆质量、规划和反馈设计、数据新鲜度，以及运行预算。系统工程常比单纯扩大 Prompt 更能决定实际表现。

LeAgent 将模型、工具、记忆、上下文配方和预算拆为独立策略，便于针对领域 Agent 分别优化；子 Agent 也按自身定义运行，而不是继承父 Agent 的全部策略。

## 1.20 Agent 的成本主要来自哪里？

主要成本是多轮 LLM 输入输出 token、长上下文重复传输、工具或外部 API 费用、代码与媒体计算、等待时间，以及失败重试和人工审核。随着循环变长，历史消息会让后续每轮更贵。

LeAgent 使用上下文相关的工具 Top-N、微压缩与自动压缩减少提示词体积，并通过 `max_turns`、`max_tool_calls_per_turn`、token 预算和运行时 profile 控制最坏成本。
