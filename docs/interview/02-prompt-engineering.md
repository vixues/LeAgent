# 二、Prompt Engineering

## 2.1 Zero-shot 和 Few-shot 的区别？

Zero-shot 只给任务说明，不给示例；Few-shot 在上下文中加入少量输入输出示例，让模型从示例归纳格式、边界和判断标准。Few-shot 通常更稳定，但会增加 token，并可能让模型机械模仿示例偏差。

实践中应先把指令和 schema 写清，再用失败样本补充少量代表性示例，而不是堆大量同质案例。LeAgent 的提示词由 `ContextManager` 按配方组装，重型领域说明采用相关性门控，避免每轮无差别注入。

## 2.2 Chain of Thought 是什么？

Chain of Thought（CoT）是让模型通过中间推理步骤处理复杂问题的方法。它有助于分解计算与逻辑任务，但公开完整隐式思维并非可靠性保证，也可能冗长、包含事后合理化或泄露敏感上下文。

生产系统更适合要求简短依据、可验证计划和工具证据，把可靠性建立在外部检查上。LeAgent 的 think-act 循环记录动作和观察，不要求把私有思维链作为系统接口。

## 2.3 Self-consistency 是什么？

Self-consistency 是对同一问题采样多条独立推理路径，再通过多数投票、评分器或验证器选择结果。它能降低单条路径偶然错误，但成本近似按采样数增长，且相关性错误不会因投票自动消失。

适合答案可比较或可验证的高价值任务；开放文本应先定义评分标准。LeAgent 可通过工作流并行运行候选分支并汇总，但这不是默认 Agent Loop 行为，应显式控制预算。

## 2.4 Prompt Injection 是什么？

Prompt Injection 是攻击者在用户输入、网页、文件或工具结果中放入指令，诱导模型忽略上层规则、泄露数据或执行越权动作。间接注入尤其危险，因为恶意文本来自 Agent 主动读取的外部内容。

防御不能只靠“不要被注入”的 Prompt，而要做指令与数据分离、最小工具权限、输入来源标记、输出校验、沙箱和高风险操作审批。LeAgent 在执行层实施 allow/deny、路径沙箱和审批策略，因此模型被诱导也不应直接获得无限权限。

## 2.5 System Prompt 和 User Prompt 的区别？

System Prompt 定义应用级身份、约束和行为优先级；User Prompt 表达本轮用户目标与数据。在支持角色优先级的模型中，system 指令通常高于 user，但它不是安全边界，仍需代码侧校验。

LeAgent 由 `PromptBuilder` 和 `ContextManager` 生成完整 system prompt，将环境、策略、记忆和相关上下文统一组装；用户消息作为独立 transcript 项进入循环。

## 2.6 为什么 Prompt 顺序会影响结果？

Transformer 对 token 位置和上下文关联敏感；靠近任务或输出位置的内容往往更容易被关注，冲突指令的先后、示例排列和长上下文中的“中间遗忘”都会改变概率分布。顺序还决定模型把某段文本解释为规则、示例还是数据。

应按“角色与硬约束—任务—必要上下文—示例—输出契约”组织，并明确分隔来源。LeAgent 用固定 context recipe 维持组装顺序，避免各调用点手工拼接产生漂移。

## 2.7 如何提高 Prompt 稳定性？

使用单一目标、明确术语、可检验约束、正反例、固定分隔符和结构化输出；降低不必要的采样随机性，缩小工具集合，并把计算与事实交给工具。还要固定模型版本并针对真实分布持续回归。

稳定性不能只靠文案。LeAgent 通过 `AgentDefinition` 固定模型温度、工具策略和运行预算，通过统一内核与上下文配方消除调用路径差异。

## 2.8 如何控制 Agent 输出格式？

从弱到强依次是：自然语言格式要求、模板与示例、JSON Mode、带 schema 的 Structured Output、应用侧解析与校验、失败后有限修复。高可靠场景必须让代码验证结果，不能只相信模型“会遵守”。

工具调用本身也是结构化输出。LeAgent 将工具参数按 schema 校验，未知键和缺失字段直接返回可操作错误；对普通最终文本则应在具体调用中使用模型支持的 `response_format` 或后置验证器。

## 2.9 Structured Output 如何实现？

先定义 JSON Schema 或类型模型，将其随请求传给支持结构化输出的模型；模型按受限解码或强指令生成，客户端再解析、校验并映射为类型。失败时只应进行有界修复或重试，并保留原始响应以便诊断。

LeAgent 的 OpenAI provider 会透传 `response_format`，支持 `json_schema` 严格模式；工具侧则以 `BaseTool.parameters` 作为权威 JSON Schema，并在执行前用 `jsonschema.validate()` 再校验一次。

## 2.10 JSON Mode 的原理是什么？

JSON Mode 通常通过模型训练、请求标志和解码约束，使输出满足 JSON 语法。它一般只保证“可解析的 JSON”，不保证字段符合业务 schema、值语义正确，也不保证不会缺字段。

因此 JSON Mode 后仍要做 schema 和业务校验。严格 Structured Output 比纯 JSON Mode 更强；LeAgent 的工具执行不会因为参数能被 `json.loads` 解析就放行，还会检查未知键、条件必填字段和完整 schema。

## 2.11 如何设计一个客服 Agent Prompt？

应包含角色与服务范围、知识来源优先级、身份与权限边界、澄清策略、退款或投诉等业务规则、何时转人工、禁止承诺事项、语气和输出格式。对账户、支付和隐私操作要要求验证身份并触发审批，而非让模型自由判断。

在 LeAgent 中可定义 `support_agent`，仅 allow `knowledge_*`、订单查询等必要工具，设置低温度和有限轮次，开启所需记忆但避免保存敏感信息；高风险工具在 `ToolExecutor` 权限层再次约束。

## 2.12 如何减少 Agent 幻觉？

要求区分事实、推断和未知；对时效性事实使用检索或数据库工具；提供可追溯证据；信息不足时先澄清；对关键结论做规则或程序验证。降低温度只能减少波动，不能消除知识性错误。

LeAgent 可通过知识检索、网页搜索和领域工具获取观察结果，并把工具成功或错误明确反馈给模型。若工具失败，应让 Agent 报告不确定性，不能把失败包装成成功。

## 2.13 如何提高 Tool Calling 准确率？

缩小候选工具集；让工具名称互斥、描述说明适用与不适用场景；schema 使用明确类型、枚举、必填项和 `additionalProperties: false`；参数名保持一致；提供必要的选择示例，并用执行错误驱动有限纠正。

LeAgent 的 `ToolRegistry` 可按 allow/deny 和上下文相关性筛选工具；参数契约规定 JSON Schema 是唯一权威定义，不提供静默别名转换，以免错误长期被掩盖。

## 2.14 Prompt 太长怎么办？

先删除重复规则和低价值示例，再把静态知识改为检索，把长历史做分层摘要，把大型工具集按任务筛选，把领域手册按相关性加载。压缩时要保留原始目标、关键约束、未完成事项和工具结果。

LeAgent 的上下文系统对重型 policy 使用 `RelevanceGate`，工具注册表支持按 query 选择 Top-N；查询循环还会执行 microcompact、渐进压缩和 autocompact，避免历史无限增长。

## 2.15 如何做 Prompt Version 管理？

把 Prompt 当代码管理：使用稳定 ID 与语义版本，保存在仓库中，变更走评审；记录模型、参数、工具 schema 和上下文配方版本；每次发布关联离线评测与线上指标，并支持回滚。仅保存最终拼接字符串不利于定位来源。

LeAgent 将人设 `prompt_variant` 与 `context_recipe` 解耦，并通过 `AgentDefinition` 引用；适合分别演进模板和上下文来源，同时用 SDK 语义化版本管理公共契约。

## 2.16 如何评估 Prompt 好坏？

先定义任务级指标：正确率、完整性、格式通过率、工具选择与参数准确率、安全违规率、拒答质量、延迟和成本。评测集应覆盖正常、边界、对抗和历史失败案例，并由确定性检查、模型评审与人工抽检交叉验证。

对 Agent 不能只评最终文本，还要评轨迹。LeAgent 的 `AgentResult`、工具调用、事件、usage 和 trace 可用于计算成功率、调用次数、耗时与错误类型。

## 2.17 Prompt A/B Test 怎么做？

随机将同分布请求分配到 A/B 版本，除 Prompt 外固定模型、采样参数、工具和预算；预先定义主指标、护栏指标、样本量和停止条件。多轮会话应按用户或会话分桶，避免同一上下文跨组污染。

上线前先离线回放，线上再小流量灰度。LeAgent 可用不同 `prompt_variant` 或 `AgentDefinition` 承载实验版本，并通过统一事件和 usage 口径比较结果。

## 2.18 如何设计多轮对话 Prompt？

应明确会话目标、已确认事实、待解决事项、引用与代词规则、何时澄清、何时结束，以及历史中用户数据与外部内容的可信级别。不要每轮重复整段历史；应保留近期原文，并对早期内容做可追溯摘要。

LeAgent 以 `TieredSessionStore` 作为聊天 transcript 真源；`AgentSession` 提供有状态 turn/stream/resume，内核检查点保存真实 messages，从而可以在等待用户输入后继续。

## 2.19 如何避免模型过度调用工具？

在 Prompt 中规定“只有缺少外部事实、需要计算或必须产生副作用时调用”，同时提供无需调用的反例；缩小工具池、设置 `tool_choice`、缓存结果，并限制轮次和调用预算。还应让模型先检查历史中是否已有同一结果。

LeAgent 支持 `tool_choice` 的 `auto`、`none`、`required` 或指定工具，并用 `max_tool_calls_per_turn`、`max_turns`、工具 Top-N 和 Agent 的 allow/deny 策略控制调用规模。

## 2.20 如何让 Agent 学会拒绝？

明确允许与禁止范围、风险分类、拒绝触发条件、可提供的安全替代方案和升级路径，并用边界案例训练或 Few-shot 演示。拒绝应简洁说明不能做什么，并尽可能帮助用户完成合法目标。

真正的拒绝能力必须由系统边界支撑。LeAgent 的 deny 规则、只读/破坏性标记、审批策略、计划模式和路径沙箱在执行层 fail closed；即使模型没有正确拒绝，越权动作也应被阻断。
