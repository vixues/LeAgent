# 四、RAG

## 4.1 什么是 RAG？

RAG（Retrieval-Augmented Generation，检索增强生成）是在生成前先从外部知识源检索相关证据，再把证据连同问题交给模型作答。典型链路是“文档解析与分块 → 建索引 → 查询检索 → 可选重排 → 上下文组装 → 基于证据生成”，核心价值是把模型参数之外、可更新且可追溯的知识引入推理。

在 LeAgent 中，知识库采用渐进式 `catalog → search → read`：`knowledge_search` 先查看文档摘要，再对 `document_chunks` 做检索，最后按命中的原文偏移读取窗口；它不会把整个知识库默认塞进系统提示词。

## 4.2 为什么需要 RAG？

模型参数知识有截止时间，无法天然掌握私有数据，也很难给出可核验出处。RAG 将“知识存储”与“语言推理”解耦，使知识可增量更新、按用户隔离、按需加载，并降低长文全部入模的成本。

LeAgent 也遵循按需注入原则：`RelevanceGate` 通过显式 opt-in、`workflow_hint` 或查询关键词决定重型领域提示是否进入上下文；知识库内容则由工具主动检索。这两者都在控制上下文预算，但前者是提示源门控，不等同于知识检索。

## 4.3 RAG 能解决幻觉吗？

不能彻底解决，只能降低“无依据生成”的概率。检索可能漏召回、错召回，文档本身可能错误，模型也可能忽略、误读或拼接证据。可靠系统还需要来源约束、引用校验、冲突处理、置信度阈值、无证据时拒答，以及端到端评测。

LeAgent 的知识库命中包含 `file_id`、`chunk_id` 和原文 `[start_offset, end_offset)`，随后回读原文窗口，有利于核验；但这不代表回答天然无幻觉。

## 4.4 Embedding 是什么？

Embedding 是把文本、图像等对象映射为稠密向量的过程，使语义相近对象在向量空间中距离更近。检索时通常用同一模型编码文档块和查询，再以余弦相似度、点积或欧氏距离排序。

要注意模型版本、向量维度、归一化方式必须一致；更换模型通常需要重建索引。LeAgent 的长期 Memory 可通过 `EmbeddingProvider` 编码查询并在 Milvus 中做语义召回；当前知识库 `knowledge_search` 本身走 BM25/词法路径，不能混为一谈。

## 4.5 Vector Database 是什么？

Vector Database 是面向高维向量存储、近似最近邻检索和元数据过滤的数据库或检索引擎。常见索引包括 HNSW、IVF，工程上还要处理分片、删除、更新、租户过滤、索引一致性与召回延迟。

LeAgent 的 Memory 在可用时使用 Milvus 向量集合；不可用时会降级到 SQL 词法检索。默认 SQLite 知识库则使用 FTS5，并不是向量数据库。

## 4.6 Chunk Size 如何选择？

Chunk Size 要在“语义完整性、检索粒度、上下文成本”之间折中。应按文档结构和任务评测选择，而不是套固定 token 数：事实问答偏小，跨段总结偏大；还要结合 embedding 模型上限、生成模型预算及 TopK。

可用 Recall@K、答案正确率、证据覆盖率和每次注入 token 数做网格实验。LeAgent 当前按字符切分，默认最多 1200 字符，并优先把完整段落装入窗口；这是工程默认值，不是普适最优值。

## 4.7 Chunk Overlap 为什么需要？

Overlap 用来保留切分边界两侧的上下文，避免实体、条件或因果链刚好被截断。它能提高边界问题的召回率，但会增加索引体积、重复命中和上下文冗余，因此通常只保留较短尾部，并在检索后去重。

LeAgent 当前将上一块最多 150 个字符的尾部带入下一块，同时保存原文偏移，便于命中后回读和核验。

## 4.8 Chunk 太大有什么问题？

块太大会混入多个主题，使向量或词法得分被无关内容稀释；命中后又占用更多上下文，降低有效证据密度，并可能让模型难以定位答案。BM25 场景下，长块还可能因词频和长度归一化表现不稳定。

LeAgent 知识库用偏移窗口回读原文，长块通常不利于精确定位；过长分块也会挤占一次检索预算内可返回的候选数。

## 4.9 Chunk 太小有什么问题？

块太小会丢失定义域、指代和前后条件，导致“关键词命中但证据不完整”。同时块数量、索引成本和重复结果上升，生成阶段需要拼接更多块，顺序恢复与去重也更困难。

因此分块应保留可回读窗口和足够上下文；LeAgent 在命中后按原文偏移扩展窗口，就是为了在小粒度索引与完整证据之间折中。

## 4.10 如何做 Semantic Chunking？

先识别结构边界，如标题、段落、列表、代码块；再计算相邻句子的 embedding 相似度，在语义突变处切分，并通过最小/最大长度约束合并过短段、拆分过长段。表格、代码和对话应使用专门解析规则，最后保留层级标题及原文偏移。

质量应通过下游检索评测验证。LeAgent 当前实现是“空行分段 + 定长装箱 + 超长段硬切”，尚不是 embedding 驱动的 Semantic Chunking。

## 4.11 TopK 如何选择？

TopK 是候选数量，不是越大越好。K 太小会漏证据，太大会引入噪声并挤占上下文。应联合召回阶段 K、重排后 K、相似度阈值和 token 预算调参，并按问题复杂度动态调整。

LeAgent `knowledge_search` 默认返回 8 条、最多 20 条；Memory recall 默认每个存储最多 4 条、合并后最多 8 条。这些是容量边界，仍需用业务数据评估。

## 4.12 Similarity Search 是什么？

Similarity Search 根据查询与候选的相似度返回最近邻。语义检索通常比较 embedding；词法检索则比较词项匹配分数。向量场景的 TopK 多为近似最近邻，因此还要关注 ANN 索引的 recall、延迟和过滤条件。

LeAgent Memory 使用查询向量在三个存储中搜索，原始余弦分数作为基础排名；若向量路径不可用或没有有效结果，则退回词法搜索。

## 4.13 Hybrid Search 是什么？

Hybrid Search 同时利用稠密语义检索与稀疏词法检索，兼顾同义表达和精确关键词。常见做法是分别召回后用 RRF 融合，或先归一化分数再加权；必须处理两套分数量纲不同的问题。

LeAgent Memory 文档称为 hybrid recall，但当前实现是各存储“语义结果为空时才词法降级”，再统一做新近度与质量重排，并非同时融合两路候选。知识库当前也只有 BM25/`ILIKE`，不要声称已实现完整 Hybrid Search。

## 4.14 BM25 是什么？

BM25 是经典概率词法排序函数，综合查询词频、文档词频逆权重和文档长度归一化。它对专有名词、编号、错误码等精确词项很强，但不理解深层语义和同义改写。

LeAgent 在 SQLite 上用 FTS5 外部内容表对文档块执行 `bm25()` 排序，并生成 snippet；FTS 不可用或在其他数据库路径上会降级为 `ILIKE` 扫描，降级路径不具备等价的 BM25 排名能力。

## 4.15 Reranker 是什么？

Reranker 对初召回候选进行更昂贵、更精确的二次排序。Cross-Encoder 会联合编码“查询—文档”，通常比双塔 embedding 更准但延迟更高；也可用 LLM 或业务规则重排。实践中先宽召回，再重排到较小 K。

LeAgent Memory 当前使用规则重排：按新近度、语义事实置信度、过程成功率、情节重要性等调整分数；知识库尚无 Cross-Encoder 或 LLM reranker。

## 4.16 Query Rewrite 是什么？

Query Rewrite 是在检索前把用户问题改写成更适合搜索的表达，例如补全上下文中的指代、提取关键词、统一术语或生成结构化过滤条件。风险是改写偏离原意，因此应保留原查询，并比较原查询与改写查询的召回结果。

LeAgent 的 `knowledge_search` 接收调用方给出的查询，没有独立的自动 Query Rewrite 管线；Agent 可以在 think-act 循环中自行换词再次搜索，但这属于代理行为，不是专门检索组件。

## 4.17 Multi Query Retrieval 是什么？

Multi Query Retrieval 为同一意图生成多个互补查询，分别检索后去重、融合和重排，以提升对措辞差异和多侧面问题的召回。它会增加检索成本，并可能引入主题漂移，所以要限制查询数并做一致性校验。

LeAgent 当前没有固定的 Multi Query Retrieval 实现；`QueryEngine` 可以根据工具结果再次调用 `knowledge_search`，但是否多查询由模型在运行时决定。

## 4.18 Graph RAG 是什么？

Graph RAG 将实体、关系、事件或社区摘要组织成图，在查询时结合图遍历与文本检索，适合多跳关系、全局主题和跨文档关联。难点在实体消歧、关系抽取质量、图更新成本和证据回溯。

LeAgent 有 ReactFlow 工作流 DAG，但那是执行编排图，不是知识图谱；当前知识库也没有 Graph RAG，二者不可混淆。

## 4.19 Agentic RAG 是什么？

Agentic RAG 让 Agent 决定何时检索、检索哪个源、如何改写查询、是否继续读取或交叉验证，而不是只执行一次固定检索。优势是能处理复杂任务，代价是延迟、成本和行为不确定性更高，需要工具权限、步数上限与轨迹评测。

LeAgent 的 `QueryEngine` 在统一 think-act 循环中可调用 `knowledge_search`，按 `catalog → search → read` 渐进取证，并依据结果继续搜索；这具备 Agentic RAG 的基本形态，但仓库没有自动保证多源验证或引用正确性。

## 4.20 Long Context 会替代 RAG 吗？

不会完全替代。Long Context 适合材料规模可控、需要全局阅读且一次性使用的任务；RAG 更适合海量、频繁更新、需权限过滤和低成本查询的知识库。长上下文仍有成本、延迟、“中间信息遗忘”和权限暴露问题，RAG 则有切分与漏召回问题。

实际常用组合是先检索定位候选，再把关键原文窗口放入较长上下文。LeAgent 正是先搜索 chunk，再按偏移回读最多受限字符的原文窗口。
